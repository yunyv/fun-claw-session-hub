# FunClaw / 主 Agent 调用 OpenClaw 的临时优化实施方案
## 1. 文档目的
这份方案不再讨论“如何把 FunClaw 做成 ArkClaw/OpenClaw 的正式 Channel 插件”。
当前目标更务实：
- 主 Agent 运行在一台独立服务器上。
- OpenClaw 运行在另一台受限服务器上。
- 由于 OpenClaw 服务器不能被外部直接访问，所以需要一台公网中转机。
- 主 Agent 需要稳定调用 OpenClaw。
- OpenClaw 需要把文本结果、图片、文件相关结果回传给主 Agent。
这是一套**临时可落地、后续可演进**的方案。
---
## 2. 当前已知条件
当前环境已经具备以下事实：
### 2.1 机器角色
1. **OpenClaw 服务器**
   - 可以主动出网
   - 外部不能直接连入
   - 上面运行 OpenClaw Gateway
2. **公网中转机**
   - 可被外部访问
   - 可以部署 sshd、反向隧道守护进程、反向代理服务、健康检查服务
3. **主 Agent 服务器**
   - 运行上层业务 Agent
   - 需要远程调用 OpenClaw
### 2.2 已经验证可用的链路
当前已经打通：
- 中转机 `127.0.0.1:28789` -> OpenClaw `127.0.0.1:18789`
- 中转机 `127.0.0.1:22222` -> OpenClaw `127.0.0.1:22`
且在中转机执行：
```bash
curl http://127.0.0.1:28789/healthz
```
返回：
```json
{"ok":true,"status":"live"}
```
这说明：
- OpenClaw Gateway 正常运行
- 反向 SSH 隧道可用
- 中转机已经可以接触到 OpenClaw Gateway
---
## 3. 先说结论
### 3.1 这个方向是对的
“OpenClaw 主动连出到中转机，再由主 Agent 通过中转访问 OpenClaw”，这个方向是对的。
它符合 OpenClaw 的推荐思路：
- Gateway 尽量只监听本机回环地址
- 不直接暴露 OpenClaw 到公网
- 远程访问优先走 SSH 隧道 / 受控代理
### 3.2 但当前结构还差半步
你现在的结构只让**中转机自己**能访问 OpenClaw：
- 因为中转机监听的是 `127.0.0.1:28789`
- 第三台机器上的主 Agent **不能直接访问**这个地址
所以要想让主 Agent 稳定接入，还需要补一个“主 Agent 到中转机”的访问层。
### 3.3 推荐优化结果
建议把方案升级成三层结构：
1. **OpenClaw 主机：只跑 OpenClaw 和反向隧道客户端**
2. **中转机：跑受控代理层，不直接裸暴露 OpenClaw**
3. **主 Agent 主机：只调用中转层提供的稳定入口**
---
## 4. 旧方案的问题
当前已打通链路是好的，但还存在几个问题。
### 4.1 主 Agent 无法直接调用
现有入口在中转机本地回环：
- `127.0.0.1:28789`
这意味着只有中转机本机进程能访问。
如果主 Agent 不部署在中转机上，而是在第三台机器上，那么它天然碰不到这个端口。
### 4.2 中转机职责不清晰
当前中转机只是“裸转发”。
这会导致：
- 没有统一入口
- 没有请求日志
- 没有接入控制
- 没有健康检查聚合
- 没有给主 Agent 提供稳定调用地址
### 4.3 `22222` 长期开着风险偏高
中转机 `127.0.0.1:22222 -> OpenClaw 22` 这条链路虽然方便，但不建议长期常驻。
因为它本质上相当于：
- 在中转机上留了一个“回打 OpenClaw SSH”的入口
哪怕只监听 `127.0.0.1`，它也应该被视为高权限调试通道，而不是长期业务通道。
### 4.4 业务接口和运维接口没有分层
目前：
- OpenClaw Gateway 业务面
- SSH 调试面
- 健康检查面
都混在一起。
后面一旦排障、扩容、加限流，会很乱。
---
## 5. 新的推荐架构
## 5.1 总体思路
采用“**反向隧道 + 中转代理 + 主 Agent 统一调用**”结构。
### 最终角色分工
#### A. OpenClaw 主机
职责：
- 运行 OpenClaw Gateway
- 只监听 `127.0.0.1:18789`
- 通过反向 SSH 主动把 Gateway 暴露到中转机本地
- 不直接对公网开放任何 OpenClaw 端口
#### B. 中转机
职责：
- 承接 OpenClaw 发起的反向 SSH 隧道
- 在本机保留内层入口 `127.0.0.1:28789`
- 再通过一个**受控代理服务**对主 Agent 提供稳定访问入口
- 做健康检查、访问控制、请求日志、连接治理
#### C. 主 Agent 主机
职责：
- 只调用中转机给出的稳定入口
- 不需要知道 OpenClaw 真正在哪台机器
- 只处理业务请求和返回结果
---

```text
│ 主 Agent 服务器     │
│                    │
│  调用统一入口       │
└────────┬───────────┘
         │
         │ HTTPS / WSS / 内网专线 / SSH 正向转发

         │
┌────────▼───────────┐
│ OpenClaw 服务器     │
│                    │
│ OpenClaw Gateway   │  127.0.0.1:18789
│ reverse-tunnel svc │
└────────────────────┘
```

## 6. 推荐实现形态
## 6.1 推荐方案分成两层
### 方案 A：最小改动版
适合马上落地。
做法：
- 保留现有反向 SSH 隧道
- 主 Agent 再主动 SSH 到中转机做一层本地转发
即：
```text
主 Agent localhost:18789
  -> SSH 到中转机
  -> 中转机 localhost:28789
  -> 反向 SSH 到 OpenClaw localhost:18789
```
优点：
- 改动最少
- 最安全
- 逻辑最简单
缺点：
- 主 Agent 也要维护 SSH 隧道
- 程序部署和自动恢复稍微麻烦一些
### 方案 B：推荐优化版
适合接业务、做临时正式环境。
做法：
- 保留 OpenClaw -> 中转机的反向 SSH 隧道
- 在中转机部署一个 `relay-proxy` 服务
- `relay-proxy` 对主 Agent 提供一个稳定的 HTTP/WS 入口
- `relay-proxy` 再转发到中转机本地 `127.0.0.1:28789`
优点：
- 主 Agent 不用自己维护 SSH 隧道
- 中转机有统一业务入口
- 方便加日志、限流、白名单、鉴权、告警
- 后面可以平滑切到 WireGuard / Tailscale / 专线
缺点：
- 中转机多了一个服务需要维护
- 需要额外做反向代理安全配置
### 本文推荐
**推荐采用方案 B。**
因为你已经明确说了：
- OpenClaw 服务器可以部署服务
- 中转机也可以部署服务
那么把中转机从“裸转发”升级成“有治理能力的接入层”，是值得的。

## 7. 新方案的详细设计
## 7.1 OpenClaw 主机设计
### 7.1.1 OpenClaw Gateway 只监听回环
要求：
- `gateway.bind: "loopback"`
- OpenClaw Gateway 只监听 `127.0.0.1:18789`
原因：
- 这样 OpenClaw 不直接暴露给公网
- 所有远程访问都通过中转层进入
- 边界最清楚
### 7.1.2 明确设置固定认证
不要依赖“启动时自动生成临时 token”。
要求：
- 显式配置固定 `gateway.auth.token` 或 `gateway.auth.password`
- 认证信息通过环境变量或配置文件管理
- 不把密钥写进代码仓库
建议：
- 对主 Agent 调用场景，优先用 `token`
- 业务程序统一走 `Authorization: Bearer <token>`
### 7.1.3 部署反向隧道守护服务
在 OpenClaw 主机部署一个常驻服务，例如：
- `openclaw-reverse-tunnel.service`
职责：
- 持续保持到中转机的 SSH 连接
- 维护以下反向映射：
  - 中转机 `127.0.0.1:28789` -> OpenClaw `127.0.0.1:18789`
- 自动重连
- 自动失败告警
建议参数：
- `ExitOnForwardFailure=yes`
- `ServerAliveInterval=15`
- `ServerAliveCountMax=3`
- `Restart=always`
### 7.1.4 `22222` 改成按需调试，不长期常驻
不建议把：
- 中转机 `127.0.0.1:22222` -> OpenClaw `22`
当作长期常驻链路。
建议改成：
- 默认关闭
- 需要排障时再临时启动
- 或单独做一个 `openclaw-debug-tunnel.service`
这样业务通道和调试通道就分开了。
---
## 7.2 中转机设计
### 7.2.1 中转机分成“内层入口”和“外层入口”
#### 内层入口
由反向 SSH 提供：
- `127.0.0.1:28789`
它只给中转机本机用，不对外暴露。
#### 外层入口
由 `relay-proxy` 提供，例如：
- `https://relay.example.com/openclaw/`
- 或 `wss://relay.example.com/openclaw/ws`
- 或中转机某个仅白名单可访问的内网地址
主 Agent 只访问外层入口。
### 7.2.2 在中转机部署 `relay-proxy`
可以选：
- Nginx
- Caddy
- HAProxy
- 一个很小的 Node/Go 反向代理服务
职责：
- 把外层请求转发到 `127.0.0.1:28789`
- 同时承接 HTTP 和 WebSocket
- 统一做安全控制
- 统一输出访问日志
- 提供健康检查聚合
### 7.2.3 `relay-proxy` 必做的事情
#### 1）只允许主 Agent 来源访问
优先级从高到低：
1. **Tailscale / WireGuard 私网访问**
2. **公网 + 安全组 / 防火墙白名单**
3. **公网 + mTLS + 白名单**
如果当前做不到私网，至少也要做到：
- 只放通主 Agent 服务器的固定 IP
#### 2）做第二层鉴权
即使 OpenClaw 自己有 token，也建议中转层再做一层。
原因：
- 防止中转层被误用
- 将来可以做多租户或多上游区分
- 能快速熔断上游问题
可选方式：
- 中转层 Bearer Token
- mTLS
- Basic Auth（不推荐长期）
#### 3）清理并补充转发头
建议中转机代理统一加：
- `X-Forwarded-For`
- `X-Forwarded-Proto`
- `X-Forwarded-Host`
并清理来路不可信的同名头。
这样 OpenClaw 可以更清楚地区分：
- 真正的主 Agent 请求
- 中转层本地请求
#### 4）提供健康检查接口
中转机应对外提供一个更靠谱的健康接口，比如：
- `/relay/healthz`
- `/relay/readyz`
其检查内容不应只看 OpenClaw `/healthz`，而应同时看：
- 反向 SSH 是否在线
- `127.0.0.1:28789` 是否可达
- OpenClaw `/readyz` 是否正常
- WebSocket 握手是否正常
### 7.2.4 中转机日志要单独保留
建议至少保留：
- 代理访问日志
- WebSocket 升级日志
- OpenClaw 上游超时日志
- 隧道存活日志
这样后面查问题时，能快速定位：
- 是主 Agent 没发出来
- 是中转机没转过去
- 还是 OpenClaw 没响应
---
## 7.3 主 Agent 侧设计
### 7.3.1 主 Agent 只对接一个稳定入口
主 Agent 不要感知：
- OpenClaw 真正在哪台机器
- 是否通过 SSH 反向隧道
- 是否切换过中转机
主 Agent 只知道一个地址，例如：
- `https://relay.example.com/openclaw/v1/responses`
- `wss://relay.example.com/openclaw/ws`
### 7.3.2 主 Agent 调用方式建议
#### 推荐首选：`/v1/responses`
适合：
- 文本结果
- 结构化输出
- 后续接工具调用
- 图片/文件输入
#### 次选：`/v1/chat/completions`
适合：
- 快速兼容现有 OpenAI 风格调用
- 以文本问答为主的场景
#### 高级控制：WebSocket RPC
适合：
- 需要 `node.invoke`
- 需要订阅事件
- 需要更细粒度控制
---
## 8. OpenClaw 能提供给主 Agent 的能力
## 8.1 文本结果
支持，最稳定。
主 Agent 可直接通过：
- `POST /v1/responses`
- `POST /v1/chat/completions`
拿回文本结果。
## 8.2 会话历史
支持。
可通过：
- `GET /sessions/{sessionKey}/history`
拉取会话历史。
适合：
- 主 Agent 做审计
- 故障追踪
- 二次处理
- 补偿逻辑
## 8.3 图片输入 / 文件输入
支持。
主 Agent 可以把：
- 图片
- base64 文件
- 指定 URL 文件
作为输入交给 OpenClaw。
## 8.4 图片输出 / 截图 / 视频结果
支持，但建议按“能力调用”方式拿，不要指望所有结果都走纯文本接口。
例如节点能力：
- `canvas.snapshot`
- `camera.snap`
- `camera.clip`
- `screen.record`
返回的数据通常是：
- `format`
- `base64`
- 宽高 / 时长 / 元数据
主 Agent 拿到后可以：
- 解码落盘
- 上传对象存储
- 回传业务系统
## 8.5 大文件结果
不建议让 OpenClaw 直接把超大二进制一路塞回主 Agent。
推荐做法：
- 小图片、小文件：直接 base64 返回
- 大文件：OpenClaw 或主 Agent 负责上传对象存储，再返回 URL
这样更稳。

## 9. 推荐的业务调用链路
## 9.1 文本调用链路
```text
主 Agent
  -> relay-proxy
  -> 中转机 localhost:28789
  -> 反向 SSH
  -> OpenClaw Gateway
  -> 执行 Agent
  -> 返回文本结果
  -> relay-proxy
  -> 主 Agent
```
## 9.2 图片/文件输入链路
```text
主 Agent
  -> relay-proxy
  -> OpenClaw /v1/responses
  -> OpenClaw 解析 input_image / input_file
  -> 执行 Agent
  -> 返回文本或结构化结果
```
## 9.3 图片/视频产出链路
```text
主 Agent
  -> OpenClaw WS / node.invoke
  -> 调用 canvas.snapshot / camera.snap / screen.record
  -> OpenClaw 返回 base64
  -> 主 Agent 解码存储
```

## 10. 安全设计
## 10.1 安全原则
这套方案的安全原则只有一句话：
**OpenClaw 不直接上公网，公网只接触中转层。**
## 10.2 必须落实的安全项
### A. OpenClaw 主机
- Gateway 只绑定 `127.0.0.1`
- 配置固定 token/password
- 不对公网开放 18789
- 反向 SSH 用独立账号、独立密钥
### B. 中转机
- `127.0.0.1:28789` 不对外暴露
- 对外只暴露 `relay-proxy`
- `relay-proxy` 只允许主 Agent 来源访问
- 做第二层鉴权
- 默认不开放 `22222`
### C. 主 Agent
- 只保存中转层调用凭据
- 不直接接触 OpenClaw 机器登录凭据
- 和 SSH 调试权限解耦

## 11. 运维设计
## 11.1 服务划分
建议最终拆成以下服务：
### OpenClaw 主机
- `openclaw-gateway.service`
- `openclaw-reverse-tunnel.service`
- `openclaw-debug-tunnel.service`（默认停用）
### 中转机
- `sshd.service`
- `relay-proxy.service`
- `relay-healthcheck.service`
### 主 Agent 主机
- `main-agent.service`
## 11.2 监控指标
建议监控：
- OpenClaw Gateway 进程存活
- OpenClaw `/readyz`
- 中转机 `127.0.0.1:28789` 探活
- `relay-proxy` 5xx 数量
- WebSocket 当前连接数
- 隧道重连次数
- 平均响应时间
## 11.3 故障定位顺序
建议固定按下面顺序排查：
1. OpenClaw Gateway 本机是否活着
2. OpenClaw -> 中转机反向 SSH 是否活着
3. 中转机 `127.0.0.1:28789` 是否可达
4. `relay-proxy` 是否正常转发
5. 主 Agent 到中转机是否正常
这样排障最快。

## 12. 分阶段落地计划
## 阶段一：当天可落地
目标：主 Agent 能稳定调用 OpenClaw。
动作：
1. OpenClaw Gateway 固定为 loopback + 固定 token
2. OpenClaw 部署反向隧道 systemd 服务
3. 中转机先不暴露裸端口
4. 主 Agent 先通过 SSH 正向转发访问中转机 `127.0.0.1:28789`
5. 先打通：
   - `/v1/responses`
   - `/v1/chat/completions`
   - `/sessions/{sessionKey}/history`
交付结果：
- 主 Agent 已经能调用 OpenClaw
- 可拿文本结果
- 可拿会话历史
## 阶段二：一周内优化
目标：中转层变成稳定入口。
动作：
1. 中转机部署 `relay-proxy`
2. 对主 Agent 提供统一 HTTPS/WSS 入口
3. 配置白名单 / mTLS / Bearer 二次鉴权
4. 增加健康检查聚合
5. 增加访问日志和告警
交付结果：
- 主 Agent 不再依赖自己维护 SSH 隧道
- 中转层具备可观测性
- 业务入口稳定
## 阶段三：后续演进
目标：从“临时方案”平滑过渡到“更正式方案”。
演进方向可选：
### 路线 A：继续保留中转层
适合：
- OpenClaw 长期放在受限网络
- 主 Agent 长期走统一出口
### 路线 B：主 Agent 与中转机改为私网直连
例如：
- Tailscale
- WireGuard
- 专线 / VPC 对等连接
### 路线 C：后续再进入插件化集成
即：
- 先把“主 Agent 调 OpenClaw”稳定跑通
- 再决定是否把 FunClaw 做成 OpenClaw / ArkClaw 正式 Channel
---
## 13. 最终推荐方案
如果只给一句建议，那就是：
### 推荐方案
- **OpenClaw 主机**：只跑 OpenClaw + 反向 SSH，Gateway 只监听 `127.0.0.1`
- **中转机**：保留 `127.0.0.1:28789` 作为内层入口，再部署 `relay-proxy` 作为外层统一入口
- **主 Agent**：只调用中转层统一地址，不直接碰 OpenClaw 主机
- **SSH 22 调试通道**：改成按需启用，不长期保留
### 这个方案的好处
1. 改动不大，能快速落地
2. 不把 OpenClaw 直接暴露到公网
3. 主 Agent 能稳定调用 OpenClaw
4. 文本、图片、文件链路都能支持
5. 后续要升级成更正式架构时，不用推翻重来
---
## 14. 适合这次项目的最短实施路径
如果要最快推进，我建议按下面顺序做：
### 第一步
先把 OpenClaw 侧固定住：
- 固定 `gateway.bind=loopback`
- 固定 `gateway.auth.token`
- 固定反向隧道 systemd 服务
### 第二步
先让主 Agent 用最小改动接入：
- 主 Agent 通过 SSH 到中转机做一层本地转发
- 先打通 `/v1/responses`
### 第三步
中转机再补 `relay-proxy`：
- 给主 Agent 一个统一稳定入口
- 补日志、白名单、健康检查
### 第四步
把图片/文件链路补齐：
- 小文件走 base64
- 大文件走对象存储 URL
---
## 15. 一句话总结
这次最优解不是“直接把 OpenClaw 暴露出去”，也不是“只靠一条 SSH 隧道硬撑”。
**最合适的临时方案是：OpenClaw 主机负责执行，中转机负责接入治理，主 Agent 只走统一入口调用。**
这样既能快，也不会把后面路堵死。
